## 2. Character & animation system (as built)

- `game/anim.ts` — `AnimController` over a `THREE.AnimationMixer`. Logical states (`idle`, `walk`, `sit`, `lie`, or any action's `animation` field value) resolve to clip names through `tuning.character.clipMap` (exact → case-insensitive → substring). Missing clips warn once and fall back to idle. Cross-fade duration, walk-clip playback rate (`walkSpeed / walkClipSpeedReference`), sit/lie perch heights — all in `tuning.character`.
- `game/world.ts` — `loadRiggedCharacter()` loads `tuning.character.meshPath`, **merges clips from every GLB in `tuning.character.animationPaths`** (Mixamo-style one-clip-per-file exports; same skeleton required), normalizes to `heightMeters` (centered, grounded), optional `yawOffsetDeg` wrapper for models not facing +Z. Load failure keeps the capsule stand-in.
- `game/sim.ts` — `onLocomotionChange` callback drives idle↔walk; `hasRig` flag disables the capsule-era tip-over lie hack; perch heights from tuning.
- `game/main.ts` — mixer advances on **sim time** (pause freezes, 2×/3× speeds up). Hot-reload re-tunes the controller and re-loads the rig when `meshPath` or `animationPaths` change.
- **`tools/animations.html` — Animation Mapper**: lists the model's actual clips, previews any clip on the character in 3D, maps clips to states via dropdowns, manages animation source GLBs, and edits all rig numbers. States shown = core (`idle/walk/sit/lie`) + every `animation` value referenced by actions + designer-added custom states. Unmapped states flagged ⚠. Saves the whole `tuning.json` (character block edited, other groups preserved).
- **T-pose diagnosis**: T-pose = no clip resolved. Open the Animation Mapper — if the clip list is empty, the animations are in separate files: add them under *Animation sources*. Otherwise the clipMap names just don't match: remap from the listed clips.
- **FBX animation sources (2026-07-14 addendum)**: `tuning.character.animationPaths` entries ending in `.fbx` (Mixamo-style: character body is one GLB, each animation is its own separate FBX) are loaded via `three/addons/loaders/FBXLoader.js` instead of GLTFLoader — `game/world.ts`'s `loadFbxClips`/`isFbxPath`/`collectBoneNames` do this, called from both `loadRiggedCharacter`'s clip-merge loop AND `tools/animations.html`'s preview loader (one implementation, per §5's never-reimplement rule). `.glb`/`.gltf` sources are completely untouched — same code path as before, byte-identical behavior when no `.fbx` path is configured. Mixamo-FBX quirks, all handled in the pure/headless-tested `game/fbxclips.ts` (`test/fbxclips.test.ts`, `npx tsx`):
  - **Bone retargeting**: each clip track is resolved against the base rig's actual bone names — exact match passes through unchanged, else the `mixamorig:`/`mixamorig` prefix is stripped and re-checked, else the track is left as-is and logged in a single one-time console warning (`retargetTrackName`/`collectBoneNames`).
  - **Scale / root motion — decision: strip, no tuning flag**. Mixamo FBX is authored in centimeters (root-position tracks ~100x off), but rather than detect-and-rescale "the root bone," ALL position tracks are dropped outright (`stripPositionTracks`) — Mixamo skeletons only ever keyframe position on the root/hip bone in the first place (everything else is rotation-only), and the game's locomotion is already procedural (`anim.ts` scales the walk clip's `timeScale` off actual ground speed, never root motion), so an in-place idle/walk/sit/lie loop has zero use for baked translation regardless of its scale. Judged unambiguous enough to not need a `tuning.character.fbxRootTranslation` flag.
  - **Clip naming**: Mixamo exports every clip embedded-named `"mixamo.com"` (or blank). `resolveClipName` falls back to the source file's basename (no extension) whenever the embedded name is missing or already used by an earlier-merged clip, with an incrementing `_2`/`_3`… suffix if even that collides — keeps the Animation Mapper's per-state clip dropdown unambiguous with multiple FBX sources.
  - Actual FBX parsing (`FBXLoader`) is browser-only, not headless-testable (jsdom can't run it) — verified instead by loading the game/tool in-browser; `game/fbxclips.ts`'s string-level decisions are what's unit-tested.
  - **Designer instructions**: drop `.fbx` files anywhere under `public/` (e.g. `public/models/anims/`), then in the Animation Mapper's *Animation sources* card add each path (e.g. `/models/anims/walking.fbx`) the same way as a GLB source — the clip(s) inside appear in *Clips in the model* and can be previewed/mapped exactly like GLB-sourced clips. No other setup needed; retargeting/scale/naming are automatic.
- The user's desired state vocabulary (idle, walking, running, sitting, laughing, talking, screaming, cooking, sleeping, reading…): `idle`/`walk` are automatic; everything else becomes live when an **action**'s `animation` field references it. "Running" would need a game-side locomotion feature (e.g. run-when-far) — not built; treat as a future tuning flag.

## 2b. Maps & the active-map mechanism (as built, Map Editor slice 2)

- **`tools/map.html`** is now the complete Map Editor (slices 1+2 in one file — note: the user's tools folder was missing the slice-1 map.html, so this file is standalone and authoritative). Modes: **Objects** (palette add, drag with the map's placement snap, R rotate with 90° normalization, inspector x/z/rot), **Floors** (drag to draw rectangle floors with a chosen material — compose several rects for L-shaped rooms; click to edit id/material/delete), **Walls** (drag to draw, axis-locked and snapped; click to edit endpoints numerically or delete), **Doors** (click a wall to place — position projects onto the wall and snaps along it, orientation inferred from wall axis; drag to move, R flips orientation), **Spawn** (click to place, facing° in inspector). Plus: **Ctrl+Z undo** (50-step snapshot stack, closes a known gap), floor-plan **underlay** image with opacity/width/offset controls (session-only tracing guide, not saved), nav overlay (game's own `nav.ts` via Vite, rebaked lazily on mutation), map-properties editing (name, gridSize, snapStep, bounds).
- **B6-6 grid refinement (2026-07-16):** `gridSize` is strictly the visual/nav/floor-cell tile size and the shipped condo now uses `0.5`m tiles; `snapStep` is a separate optional map field (shipped `0.25`, default `0.25` for old maps) used by all Map Editor placement gestures. Coordinates and asset footprints remain meters and were not rescaled. New maps start with the same 0.5/0.25 pairing.
- **Multiple maps**: header select + **New / Duplicate / Delete** (new & duplicated maps are PUT to `data/maps/<id>.json` immediately). `server.js` gained `GET /api/maps` (listing) and `DELETE /api/data/maps/*.json` (deletion is refused for non-map files). Deleting the active map or the last map is blocked in the tool.
- **Active map**: `tuning.map.active` names which `data/maps/<id>.json` the game plays (default `condo`). `data.ts` loads tuning first, then that map. The editor's **"▶ Play this map"** button PUTs tuning with the current map id — the running game hot-reloads into it, and `main.ts` detects the map-id change and teleports the sim to the new spawn (`SimAgent.teleportTo`).

## 2c. Needs & skills add/remove (as built, Tuning Editor extension, 2026-07-13)

- **B5-1 non-linear growth (2026-07-16):** `SimStats.applyGains` remains the single skill-delta chokepoint. Positive action gains are now scaled by `(1 - clamp(level/max, 0, 1)) ^ tuning.skills.growthCurveExp` (default `1.5`); exponent `0` restores linear gains, max-level gains are zero, and negative loss/decay deltas are unchanged. The pure `scaleSkillGain` helper is covered by `test/stats.test.ts`; tuning hot-reloads with the rest of `tuning.json`.
- **No new tool file** — `tools/tuning.html`'s existing Needs/Skills sections already render one editable group per entry (`renderNeeds`/`renderSkills`), so add/remove was a natural extension rather than a new Stats Editor page. A dedicated `tools/stats.html` would have duplicated that machinery for no benefit; extending was the cleaner call here (contrast with the Map Editor, which got its own file because it needed a canvas + very different interaction model).
- **Add**: "+ Add need" / "+ Add skill" toolbar buttons (`[data-action="add-need"|"add-skill"]`) at the top of each section. `window.prompt` asks for a display **name only**; the id is derived (`slugify` → lowercase, non-alphanumerics → `_`) and uniquified against the existing list (`thirst`, `thirst_2`, …) — no separate id prompt, no failure path to retry. New-entry defaults are **averaged from existing (non-computed) entries**, not hardcoded: a need copies the mean `default`/`decayPerTick` of current needs and starts `autonomy: true`; a skill starts at `default: 0` (skills grow through practice, never decay — no `decayPerTick` field exists on `SkillDef`) with `max` averaged from existing skills. Bar `color` is randomized (designer edits it via the existing color field same as any entry).
- **Remove**: per-entry "Delete need"/"Delete skill" buttons (`[data-action="delete-need"|"delete-skill"][data-need-id|data-skill-id]`). Referential integrity is checked against `interactions.json`: a need can be referenced by an action's `needGains` key or its `primaryNeed`; a skill only by `skillGains`. **Chosen UX (deliberately different from the Asset Editor's "warn and leave dangling" pattern)**: the confirm dialog lists every referencing action by name and field, and on confirm the tool **strips the dangling references** (deletes the `needGains`/`skillGains` key, nulls a matching `primaryNeed`) in the same save cycle — `interactions.json` is marked dirty alongside `stats.json` so both PUT together. Rationale: `game/stats.ts`/`game/autonomy.ts` already no-op safely on an unknown id (nothing crashes either way), but shipping a `needGains`/`skillGains` key that points at nothing is exactly the kind of orphaned data this data-driven design is supposed to avoid, and stripping is a single confirm click away from being wrong-footed by a docs/CLAUDE.md-style comment describing a field that no longer means anything.
- **Ordering gotcha (documented in code comments)**: `render()` fully rebuilds section DOM (`innerHTML = ''` + re-append), which discards any `.dirty` class applied beforehand — `markDirty(file)` must be called **after** `render()` in the add/remove handlers, not before, or the dirty indicator (and the Save button's enabled state) silently fails to reflect the change. (Plain field edits don't hit this because they mutate in place without re-rendering.)
- **No hardcoded ids found in game code**: audited `game/*.ts` for literal need/skill strings (`hunger`, `energy`, `english`, etc.) — none exist. `game/stats.ts` (`SimStats`), `game/ui.ts` (`Hud.rebuildBars`), and `game/autonomy.ts` all iterate `stats.needDefs`/`skillDefs`/`lowestAutonomyNeed()` purely by whatever ids are in `stats.json`; the Environment "computed" need is found dynamically (`data.stats.needs.find(n => n.computed)` in `game/main.ts`), not by id. Confirmed live: added/removed needs and skills render on the in-game HUD and drive decay/autonomy with zero code changes.
- **Tests**: `tools/tuning-editor.test.mjs` extended (not a new suite) with add/cancel/uniquify coverage for both needs and skills, and both referential-integrity branches (unreferenced → plain message; referenced → lists the action(s), strips the reference on confirm, cancel leaves everything untouched), plus a final PUT assertion proving `stats.json` and `interactions.json` save the fully-reconciled state together.
- **Environment note**: this session also discovered the repo's `node_modules` had never actually been installed (no `jsdom`/`tsx`/`canvas`) — every existing test suite would have failed to even start. Added `jsdom`, `tsx`, and `canvas` as devDependencies so the full suite (tool `.mjs` + game `.ts`) is runnable; this is a one-time environment fix, not a code change.

## 2d. Interaction Editor (as built, 2026-07-13)

- **New tool file `tools/interactions.html`** — sidebar (all actions, "player-only" badge when `autonomyEligible` is false) + editor panel, following the Asset Editor's layout (contrast with §2c's needs/skills, which extended an existing section instead — actions have enough fields and their own referential-integrity direction that a dedicated page was the cleaner call here, same reasoning as why Map Editor got its own file).
- **Autonomy-eligibility flag: pre-existing, not added.** `data/interactions.json`'s `ActionDef` already carried `autonomyEligible: boolean` on every action (`game/data.ts` `ActionDef` interface, line ~13), and `game/autonomy.ts`'s `maybeAct()` already gates candidates on it (`if (!action || !action.autonomyEligible) continue;`) before matching `primaryNeed`. No schema change and no `autonomy.ts` change were needed — the editor just exposes the existing flag as a checkbox, so e.g. "Wash hands" or "Cook" (already `autonomyEligible: false` in the shipped data) stay player-only, and toggling it live-round-trips through the sim's autonomy search on save.
- **Editable per action**: name, free-text `animation` state (datalist + hint line listing every state already in use — the 4 core states `idle`/`walk`/`sit`/`lie` the Animation Mapper always recognizes, plus every other action's current `animation` value — so the designer isn't guessing a name that already drives a mapped clip), `autonomyEligible`, `seatAware` (sparse — key only present when true, mirroring the Asset Editor's `seatTarget` convention), `primaryNeed` (dropdown fed from `stats.json` needs + "(none)", writes `null` not a missing key — matches the schema's `primaryNeed: string | null`), and one numeric row per need/skill in `stats.json` for `needGains`/`skillGains` (blank = no entry in the map) — same "fed from data, no free-typed ids" pattern as the Tuning Editor's per-action gain rows, just given a dedicated full-field page instead of a compact grid.
- **id is fixed** once created (assets reference it by string, like an asset's own id) — editing swaps the display name only.
- **Add**: "+ New action" prompts for a display name only; id = `slugify(name)` uniquified against existing action ids (`do_yoga`, `do_yoga_2`, …) — identical derivation helper to the Tuning Editor's `addNeed`/`addSkill`. Defaults: `needGains: {}`, `skillGains: {}`, `animation: ''`, `autonomyEligible: true`, `primaryNeed: null` (sane no-ops until the designer fills them in).
- **Referential integrity** is checked against `data/assets.json` (an asset's `interactions: string[]` array). Delete follows the same "warn with names, strip on confirm" policy as §2c: unreferenced action → plain confirm; referenced action → confirm dialog lists every referencing asset by name, and on confirm the tool removes the id from those assets' `interactions` arrays and deletes the action, marking **both** `interactions.json` and `assets.json` dirty so they save together in one PUT cycle — `assets.json` never ships pointing at an action id that no longer exists.
- **Hot-reload**: works with zero changes. `game/data.ts`'s `watchData()` already polls and reloads the whole `GameData` bundle (including `interactions.json`) every 2s via `loadAll()`, and `game/main.ts`'s hot-reload callback does `data = fresh` (a `let`, not `const`) — every consumer that reads `data.interactions.actions` at use-time (e.g. building the action menu, `WatchTick`-equivalent gain application, `autonomy.ts`'s `actionsById` map) picks up edited/added/removed actions on the next poll with no extra plumbing.
- **Tests**: new suite `test/interaction-editor.test.mjs` (placed alongside the other newest suites, `test/map-editor.test.mjs` and `test/animation-editor.test.mjs`, rather than `tools/` where the two oldest suites still live) covers render + player-only badge, editing every field including the sparse `seatAware` key and the `primaryNeed` null round-trip, add with slugify+uniquify, both delete branches (unreferenced plain-confirm; referenced confirm-lists-assets-and-strips, asserting the exact stripped `assets.json` PUT payload), and search filtering.

## 3. Quest system — locked design (build in a future slice)

Replaces the roadmap's vaguer "Story Editor". Two parts:

### 3.1 Sim-state variables (`data/simstate.json` or a `variables` block)
Designer-defined variables beyond needs/skills, e.g. `visaStatus` (string), `job` (string|null), `income` (number), plus the built-in `funds`. Defaults live in data; the save system persists current values. Variables are readable by quest conditions and writable by quest rewards. This also finally gives `tuning.economy.startingFunds` a consumer.

### 3.2 Quests (`data/quests.json`) — draft schema
```json
{
  "quests": [{
    "id": "first_words",
    "name": "First Words",
    "description": "Reach English level 10 and save §500.",
    "trigger":    { "all": [ { "var": "skills.english", "gte": 1 } ] },
    "completion": { "all": [ { "var": "skills.english", "gte": 10 },
                             { "var": "funds", "gte": 500 } ] },
    "rewards": [ { "type": "funds", "amount": 200 },
                 { "type": "setVar", "var": "visaStatus", "value": "student" } ],
    "onceOnly": true
  }]
}
```
- **Condition namespace**: `needs.<id>`, `skills.<id>`, `funds`, `time.hour`, `time.day`, `vars.<name>`, `quests.<id>.state` (`locked|active|done`).
- **Operators**: `gte`, `lte`, `eq`, `neq`; combinators `all` / `any`, nestable.
- **Reward types**: `funds`, `setVar`, `unlockAsset` (hides/shows catalog entries — ties into Buy/Sell mode), later `dialogue` / vocabulary popups for the language-learning content.
- **Evaluation**: on the needs-decay tick (same reuse-an-existing-interval convention as autonomy). Quest state persists in the save.
- **Quest Editor tool**: CRUD quests, condition builder with dropdowns fed by `stats.json`/variables (no free-typed ids), live validation, and a quest log HUD panel in-game (trigger → "quest started" toast; completion → rewards applied + toast).

Build order within the quest slice: 1) variables + condition evaluator (headless-tested — this is pure logic, perfect for the test suite), 2) in-game quest runner + HUD log, 3) Quest Editor tool.

## 3.3 Quest system runtime — as built (steps 1–2, 2026-07-13)

Steps 1–2 of §3's build order are shipped (game side only — no Quest Editor tool yet, that's step 3).

- **New data files**: `data/simstate.json` (designer variable defaults — `visaStatus` string, `job` string|null, `income` number, each `{ id, name, type, default }`; `funds` is deliberately NOT in this file, see below) and `data/quests.json` (shipping the exact `first_words` example from §3.2, ids matched to the real `stats.json`: `skills.english` — English's `max` is 10, matching the quest's `gte: 10` completion threshold). Both are generic `data/*.json` files; `server.js`'s `/api/data/<path>` and `/data/<path>` routes already serve/accept any `.json` under `data/` with no code change needed — verified by loading both through the existing `GET/PUT` handlers.
- **New module `game/quests.ts`** (pure, no DOM/three.js — headless-testable):
  - `evaluate(condition, ctx)` — recursive evaluator for the §3.2 condition tree (`all`/`any`, `gte`/`lte`/`eq`/`neq`) against an `EvalContext` snapshot (`needs`, `skills`, `funds`, `time.{hour,day}`, `vars`, `quests` — all plain records, built fresh each tick by the caller).
  - **Unknown-id semantics (decision)**: any `var` path that doesn't resolve (unknown need/skill/var/quest id, or an unrecognized namespace) resolves to `undefined`, and every operator against `undefined` is `false` — never throws. A leaf with none of `gte/lte/eq/neq` set is likewise `false`. `all([])` is vacuously `true`; `any([])` is `false` (nothing to satisfy) — both just `Array.prototype.every`/`some`'s native behavior, called out explicitly in the code comment as intentional.
  - `QuestRunner` — the runtime state machine: `quests: Record<id, 'locked'|'active'|'done'>`, `vars: Record<id, value>`, `funds: number`, `unlockedAssets: Set<string>`, `completedLog: string[]`. `tick(needs, skills, time)` evaluates all `locked` quests' triggers (→ `active`, fires `onQuestStarted`), then all `active` quests' completions (applies rewards, fires `onQuestCompleted`) in the same pass — a quest can in principle trigger and complete in the same tick if its completion condition already holds; documented as intentional rather than special-cased, since §3.2 doesn't require a one-tick delay.
  - **onceOnly semantics**: `onceOnly: true` quests go `active → done` and stay `done` forever (rewards apply exactly once). `onceOnly: false` quests go `active → done` (rewards apply) then **immediately reset to `locked`**, so they can trigger again later — this is the "repeatable quest" reading of §3.2's "onceOnly:true quests never re-trigger" (which implies the `false` case can). Not explicitly speced beyond that phrase; flagged here as a design call for the Quest Editor agent to know about (a repeatable quest's rewards fire every time it completes).
  - **Rewards**: `funds` (add to balance), `setVar` (write `vars[name] = value`), `unlockAsset` (add an asset id to `unlockedAssets`).
  - **unlockAsset / Buy-Sell tie-in**: implemented as an additive `Set<string>` of asset ids a quest has explicitly unlocked (`isAssetUnlocked(id)` helper), not a schema change to `assets.json` — no asset currently opts into being quest-gated. Documented expectation for the future Buy/Sell mode (roadmap §4 item 6): an asset is purchasable if it has no quest-gate flag OR `isAssetUnlocked` is true; that gate flag itself (e.g. a future `AssetDef.requiresQuestUnlock`) is for that slice to add.
  - **Hot-reload (`retune`)**: adopts new/edited quest and variable *definitions* without touching runtime *state* — existing quest states and variable values are preserved; only brand-new quest ids get `locked` and brand-new variable ids get their default. Removed ids leave an orphaned (harmless) runtime entry, mirroring `stats.ts`'s `retune()` convention.
  - **Persistence — deferred**: there is no save system in this repo (`data/save/` is empty and gitignored; no `save.ts` module exists anywhere). Quest/variable/funds state is therefore **runtime-only** (resets on page reload), same as needs/skills today. `QuestRunner.serialize()`/`restore(QuestSaveState)` produce/consume a flat JSON-serializable shape specifically so wiring a future save system is a direct `JSON.stringify(runner.serialize())` / `runner.restore(parsed)` call, no refactor needed.
- **`game/data.ts`** gained `VarDef`/`SimStateData`, `Condition`/`ConditionLeaf`/`ConditionAll`/`ConditionAny`, `QuestState`, `Reward`/`RewardFunds`/`RewardSetVar`/`RewardUnlockAsset`, `QuestDef`/`QuestsData` types; `GameData` gained `simstate`/`quests`; `TuningData` gained `quests: { toastDurationSeconds, completedLogLimit }` (both now in `data/tuning.json`, no hardcoded numbers); `loadAll()` fetches the two new files alongside the rest.
- **`game/main.ts`** wiring: `QuestRunner` constructed after `stats`/`hud`/`autonomy`, seeded with `data.tuning.economy.startingFunds` (giving that field its first real consumer, per §3.1). `quests.tick(...)` is called **inside the existing needs-decay `while` loop** in `simTick` (no new accumulator/timer — reuses `decayAcc`/`needsDecayTickSeconds`, per §3.2's "same reuse-an-existing-interval convention"). Needs/skills are snapshotted per-tick via `Object.fromEntries(stats.needs/.skills)`. `time.day` is new: the render loop's clock advance changed from `gameSeconds = (gameSeconds + …) % 86400` to an explicit `while (gameSeconds >= 86400) { gameSeconds -= 86400; gameDay++; }`, so `time.day` (1-indexed, starts at 1) is now tracked and available to conditions. Quest start/complete callbacks trigger HUD toasts and rebuild the quest log; hot-reload calls `quests.retune(...)` + rebuilds the log.
- **HUD (`game/ui.ts`)**: new collapsible `#quest-panel` (bottom-left, above the devbar, same visual language as the needs/skills panels) listing active quests (name + description) and a "Completed" section (last `tuning.quests.completedLogLimit` entries, newest first). ## 3.4 Quest Editor tool (as built, 2026-07-13)

Step 3 of §3's build order — the last piece of the quest slice. **New tool file `tools/quests.html`**, no game-module imports needed (pure JSON editing, so it's a plain inline script only, same as `tools/tuning.html` — no `type="module"` section, contrast with the Map Editor / Animation Mapper which genuinely need `nav.ts`/`world.ts`).

- **Layout**: one page, two `<section>`s stacked (mirrors `tools/tuning.html`'s multi-section single-column layout, not `tools/interactions.html`'s single sidebar+editor, since this tool has two distinct entity kinds). **Variables** (`data/simstate.json`): a toolbar (`+ Add variable`) + one row per `VarDef` — name, id (fixed after creation), a `type` dropdown (`string`/`number`/`boolean`, matching the real `VarDef.type` in `game/data.ts` — not the draft `string|null` phrasing in the slice brief, which the shipped type superseded), and a default-value control with a separate "null" checkbox (needed because `VarDef.default` can be `string | number | boolean | null` independent of `type` — e.g. the shipped `job` variable is `type: 'string'` with `default: null`). Changing `type` coerces any non-null default to a sane value of the new type. **Quests** (`data/quests.json`): a sidebar (name + a "repeatable" badge when `onceOnly` is false) + an editor panel, same visual pattern as `tools/interactions.html`.
- **Condition builder**: recursive renderer over the `Condition` tree (`isLeafCond`/`renderGroupBox`/`renderLeafRow`). Every rendered node carries a `data-cond-path` positional address (`"trigger"`, `"trigger.0"`, `"trigger.1.0"`, …) computed from array indices at render time; every structural edit (add leaf, add group, remove, change combinator, change a leaf's var) triggers a full `renderQuestEditor()` re-render (paths are recomputed fresh each time — simpler and more robust than trying to preserve DOM identity across array-length changes, at the cost of losing focus on structural edits; plain value typing does **not** re-render, so text/number entry keeps focus). Each group box has an All/Any `<select>` (swapping the object's `all`/`any` key while keeping the same child array) plus inline hint text stating the empty-group semantics; each leaf is a var-path `<select>` (dropdown-only, **never free-typed** — options are grouped by namespace: Needs/Skills from `stats.json`, Economy (`funds`), Time (`time.hour`/`time.day`), Variables from `simstate.json`, Quests (`quests.<id>.state`, self-reference allowed per the brief) + an operator `<select>` (`gte`/`lte`/`eq`/`neq`) + an adaptive value input resolved by `describeVarPath()`: number input for needs/skills/funds/time, a `locked`/`active`/`done` `<select>` for quest-state paths, and for `vars.<name>` paths a control typed by that variable's own `type` (checkbox-as-select for boolean, number input for number, text otherwise). A stale/hand-edited var path that doesn't match any live option still renders (in an "Unknown (stale id)" `<optgroup>`) rather than silently snapping to a different value. Changing a leaf's var or operator resets its value to a type-appropriate default (`defaultValueFor`), never leaving a type-mismatched leftover (e.g. a string value under `gte`).
- **Rewards builder**: one row per `Reward`, a type `<select>` (`funds`/`setVar`/`unlockAsset`) that swaps in a fresh default reward object on type change, then type-specific fields — `funds`: amount number input; `setVar`: variable dropdown (from `simstate.json`) + adaptive value input (same typing rule as condition leaves); `unlockAsset`: asset dropdown (from `assets.json`). Add/remove rows via `+ Add reward` / a per-row `× remove`.
- **Add quest**: name prompt → `slugify` + `uniqueId` (identical helper pattern to every prior editor) → `{ trigger: {all:[]}, completion: {all:[]}, rewards: [], onceOnly: true }`. The empty-`all` vacuous-TRUE-on-both-trigger-and-completion warning isn't a separate dialog — it shows up immediately in the inline group hint text *and* the validation panel, which is arguably more useful (visible for the life of the quest, not just at creation).
- **Referential integrity — both directions, same "warn with names, strip on confirm" policy as §2c/§2d**:
  - **Delete a variable**: scans every quest's `trigger`/`completion` trees for a leaf with `var === "vars.<id>"` (via a generic `anyLeaf`/`stripCondition` recursive walker reused for both variables and quests) and every `rewards` array for a `setVar` entry with that `var`. Confirm dialog lists referencing quests by name and which part references it; on confirm, condition leaves are stripped (the containing `all`/`any` array is kept, even if now empty — an emptied `all` becomes vacuously TRUE, an emptied `any` becomes FALSE, both stated in the dialog text) and matching `setVar` rewards are removed outright (there's no meaningful "null" reward to leave behind). Both `simstate.json` and `quests.json` are marked dirty and saved together.
  - **Delete a quest**: scans every *other* quest's conditions for a leaf `var === "quests.<id>.state"` (self-references are irrelevant since the quest and its own leaf disappear together). Same strip-on-confirm behavior, `quests.json` only (a quest reference never touches `simstate.json`).
  - `stripCondition(node, predicate)` mutates combinator arrays in place, recursing through arbitrarily nested `all`/`any`; it returns `null` only if the *root* condition is itself a stripped leaf (bare-leaf root is legal per the `Condition` union though the editor never produces one), and callers fall back to `{ all: [] }` in that case.
- **Validation panel**: recomputed on every render (`collectValidationIssues` → `renderValidation`), covers (a) empty `all`/`any` groups with their exact runtime semantics spelled out, (b) unknown ids for every namespace (`needs.`/`skills.`/`vars.`/`quests.<id>.state`) plus unrecognized namespaces entirely, (c) a leaf with none of `gte`/`lte`/`eq`/`neq` set, (d) reward-side unknown ids (`setVar`'s var, `unlockAsset`'s asset) and a non-numeric `funds` amount. **Decision: validation never blocks saving** — the header text says so explicitly ("warnings only, saving is never blocked"), because a designer mid-edit (e.g. a quest with an intentionally empty group while wiring up its conditions) shouldn't be locked out of saving their in-progress work; the dropdowns already prevent *new* unknown ids from being created; unknown ids only show up from hand-edited JSON or an id deleted elsewhere, and the panel's job is to surface that, not gate it.
- **Tests**: new `test/quest-editor.test.mjs` (modeled on `test/interaction-editor.test.mjs`) — render (variable rows, quest sidebar incl. repeatable badge), variable CRUD (type/default edits incl. the null-default toggle, add+uniquify, both delete branches with exact stripped PUT payloads spanning both files), quest add+uniquify, a full condition-builder sequence building a nested `{all:[{var:funds,eq:500},{any:[{var:skills.english,gte:5}]}]}` tree and asserting the *exact* resulting JSON, leaf removal, all 3 reward types (add/edit/remove) with exact PUT payloads, quest delete both branches (referenced — strips the other quest's condition leaf; unreferenced — plain confirm), validation-panel assertions (unknown need/variable ids, missing operator, emptied group semantics), and sidebar search filtering. ### 7.17 Bladder failure — "pees itself" (ROADMAP_NEXT B2-4, 2026-07-15)

- **Trigger/cooldown, pure logic**: new module `game/bladder.ts` (no THREE/DOM dependency, mirrors `game/duration.ts`'s split), headless-tested in `test/bladder.test.ts` (20 assertions, `npx tsx`). `BladderFailureState { armed: boolean }` is a simple latch, not a time-based cooldown — `stats.ts`'s `decayTick` clamps bladder at 0 (never negative), so it can sit AT 0 for many consecutive decay ticks once it bottoms out; without a latch the event would refire every tick while low. `checkBladderFailure(state, bladderValue, reliefAmount)`: fires (and disarms) the moment bladder reaches `<= 0` while `armed`; re-arms only once bladder climbs STRICTLY above `reliefAmount` (not just above 0) — the relief refill itself lands exactly AT `reliefAmount`, so a plain re-decay from there back to 0 with no real bathroom visit must not retrigger; only a genuine need-satisfying top-up (autonomy/player using the toilet, which fills well past `reliefAmount`) re-arms it.
- **Tuning**: `TuningData.bladderFailure?: { durationSeconds?: number; reliefAmount?: number }` (`game/data.ts`, sparse/optional — same precedent as `fire?`/`garbage?`). Shipped `data/tuning.json`: `{ durationSeconds: 4, reliefAmount: 30 }` (the roadmap's own defaults, verbatim). `game/main.ts` falls back to the same `{4, 30}` if the whole block (or a field) is absent.
- **Event wiring, `game/main.ts`**: the zero-crossing check runs in `simTick`'s needs-decay loop, immediately after `stats.decayTick()` and BEFORE `autonomy.maybeAct()` on the same tick — so a fresh failure preempts whatever free will would otherwise have picked that tick, matching "the event preempts everything." On fire, `triggerBladderFailure()`: (1) `agent.stopAction()` — interrupts whatever the sim was doing, fires the normal `onActionStop` path (animation reset to idle, activity chip hidden, any accident roll/waste handling for the interrupted action, exactly as any other stop reason); (2) `autonomy.forceCooldown(durationSeconds)` — new `Autonomy` method (distinct from `notePlayerCommand`'s fixed tuning-driven cooldown) that suppresses free will for an EXPLICIT duration, so autonomy can't immediately walk the sim off toward the toilet the instant `agent.isBusy` goes false mid-"pee" animation; (3) `anim?.play('pee')`; (4) `accidents.spawnTransient('pee_puddle', [sim.position.x, sim.position.z], rotDeg, simClockSeconds)` — the sim's EXACT current position/facing, reusing §7.3/item-10's existing spawn mechanics with no changes needed there. A standalone `peeState: { elapsed, totalSeconds } | null` closure timer (NOT tied to an `ActiveAction` — this event has no target asset, so `sim.ts`'s action system doesn't apply) ticks on the same sim-time `sdt` as `durationState`/doors/the animation mixer via the SAME `isDurationComplete` helper from `game/duration.ts` (§7.11 reuse, no new timer math); on completion it plays `'idle'` and calls a new `SimStats.refillNeed('bladder', reliefAmount)` — an absolute set (not a per-tick delta like `applyGains`), guarded the same way `setComputed` guards its writes. Once `forceCooldown`'s window lapses, autonomy resumes normally on its own (per the roadmap note: "bladder at 30 may immediately seek the toilet — fine, that's Sims behavior").

## 7. Locked designs — designer requests of 2026-07-14

Four requests from the designer, specced here before implementation (build order in §4 items 6–9). Common thread: everything is data + tool UI per the §1 rule.

### 7.1 Doors as their own asset type

- **Doors become entries in `assets.json`** with `category: "door"` — they reuse the whole asset machinery (mesh, footprint, editor) instead of a parallel system. A new general field `buyable: boolean` (default `true`, absent = true) is added to `AssetDef`; door entries ship `buyable: false` — the designer doesn't want players buying/selling doors *yet*, but flipping one flag changes that later. The future Buy/Sell catalog must filter on `buyable`.
- **Mesh-fit fields, general to ALL assets** (any imported GLB may come mis-scaled/mis-oriented): optional `meshFit: { scale?: number | [x,y,z], yawOffsetDeg?: number, yOffset?: number }` on `AssetDef`, applied by the loader after normalization. This answers the door "resize as required" request but deliberately lands on every asset.
- **Door-specific block** on door-category assets: `door: { hingeOffset: [x, z], openAngleDeg, openSeconds, closeSeconds, triggerDistance }`. `hingeOffset` is the rotation-axis position in model-local meters relative to the model origin — the designer's requested movable Z-rotation axis, so any GLB can be made to swing correctly regardless of where its author put the origin. Per-asset values override tuning defaults `tuning.doors: { openSeconds: 0.5, closeSeconds, openAngleDeg: 90, triggerDistance }` (all live-tunable; 0.5s is the designer's requested default).
- **Map link**: a map's `doors[]` entries gain an optional `assetId` (a door-category asset). With an `assetId` the game renders that asset's mesh on the doorway and animates it; without one, current behavior (bare opening) is unchanged — old maps stay valid. The Map Editor's Doors mode gets a door-asset dropdown (door-category assets only) in its inspector.
- **Open/close logic** (mirrors the Unreal `BP_Door` semantics, which are PIE-proven): each frame-ish (reuse an existing tick, no new timer): if the sim is within `triggerDistance` **and** its current nav path crosses the doorway segment → rotate open around the hinge over `openSeconds`; once the sim is farther than `triggerDistance`, rotate closed over `closeSeconds`. Never close while the sim is within the trigger distance (never trap/clip the sim mid-transit). Standing near a door without a path through it does NOT open it.

### 7.1 as-built (2026-07-14)

- **New module `game/doors.ts`** (pure math, no THREE dependency in its logic half — headless-tested in `test/doors.test.ts`, following the `game/facing.ts` precedent of "dedicated module with pure-testable logic + a thin three.js layer"): `resolveDoorConfig` (sparse per-asset `door` block merged over `tuning.doors`, `null` when the asset has no `door` block), `doorBaseYawDeg`, `hingeWorldPos`/`panelLocalOffset` (hinge transform math), `segmentCrossesDoorway`/`pathCrossesDoorway` (the pair-crossing test), `distanceToDoor`, `doorShouldBeOpen` (the open/close state machine), `stepDoorAngle` (linear swing timing), `isAnimatedDoor`. The three.js layer (`createDoorInstance`/`buildDoors`) turns that into a rotating hinge pivot + panel mesh, reusing `world.ts`'s `loadMeshTemplate`/`applyMeshFit`/`normalizeModelToFootprint` (now exported) rather than reimplementing GLB loading — same "instant stand-in box, async GLB swap, keep the box on failure" pattern as furniture.
- **Tick choice**: no new timer. `main.ts`'s existing render loop calls `doorInstance.update(sdt, simPos, simPath)` once per frame, right after `agent.update(sdt)`, using the SAME sim-time `sdt` the animation mixer advances on — pausing the game freezes doors mid-swing, 2×/3× speeds them up, exactly like the character's animation. `simPath` comes from a new `SimAgent.getPathPoints()` (`game/sim.ts`): current position + remaining unvisited waypoints, empty while not moving — the "current nav path" the crossing test needs.
- **Hinge convention**: `AssetDef.door.hingeOffset` is defined in the door's CANONICAL model-local frame — local **+X is the door's long/swing axis**, local **+Z is its thickness axis** — the same frame regardless of which way the door is oriented in the map. Placing a door rotates that frame by a fixed "base yaw" so its long axis lies along the wall it sits in: `'horizontal'` orientation (wall runs along world X) → base yaw 0°; `'vertical'` orientation (wall runs along world Z) → base yaw 90°. This exactly reproduces the dimensions of the OLD plain-marker box world.ts already drew per orientation (`BoxGeometry(1.0,h,0.14)` horizontal / `BoxGeometry(0.14,h,1.0)` vertical) — same shape, now parented under a hinge pivot instead of centered on the doorway. The panel's local offset from the hinge is `-hingeOffset` (same canonical frame), so the closed panel's center exactly reproduces the door's original `at` position — verified analytically in `test/doors.test.ts` for both orientations, not just by construction.

### 7.2 Asset facing direction (the "normal" vector)

- `AssetDef` gains `facingDeg: number` (default 0): the direction the object faces in **model-local space**, expressed as a yaw in degrees (0 = the model's local −Z after `meshFit.yawOffsetDeg`, i.e. what the game already treats as "toward the camera at rot 0" — the implementing agent must verify and document the exact convention against `world.ts`'s placement math). An instance's **world facing** = instance `rot` + `facingDeg`. A single yaw is exactly the "normal vector telling me where the object is facing" the designer asked for, in the friendliest editable form (degrees, not a vector triple) — tools may *display* it as an arrow.
- **Consumers (game)**: a shared `useSpotFor(instance)` helper — the interaction stand/approach point becomes "footprint edge along world facing + small clearance" instead of any current pivot/center heuristic (fridge opens from its front, WC approached from the front…); seat-aware logic ("sit in front of the TV") screens candidate seats by the half-space test `dot(seatPos − tvPos, tvFacing) > 0` and the seat's own facing toward the target — this replicates the Unreal Phase-A seat-in-front-of-screen logic that needed a hardcoded RightVector exception for the TV mesh; with `facingDeg` per asset that exception class disappears.
- **Tools**: Asset Editor gets the `facingDeg` field (and `meshFit`, and `buyable` from §7.1); Map Editor draws a small facing arrow on every placed object (read-only indicator, instance rot + asset facingDeg) so the designer can see at a glance which way furniture faces.

### 7.3 Accidents

- **Accident types are assets too**: `category: "accident"`, `buyable: false` (fire, water puddle, debris…). They get mesh/footprint/interactions through the existing Asset Editor for free. Their `interactions` are the cleanup actions (mop, extinguish — authored in the Interaction Editor like any action); their `environmentScore` should typically be negative while present. New field on accident-category assets: `clearedBy: string[]` — action ids whose completion on the accident instance despawns it.
- **Per-asset risk config** on any normal asset: `accidents: [{ accidentId, trigger: "onUse", baseChancePercent, placement: "on" | "adjacent", adjacentRange: [1, 2], modifiers: [{ var, pctAt0, pctAtMax }] }]`.
  - **Roll timing**: once, when a sim *finishes* an action on the asset (`trigger: "onUse"` is the only trigger for now; the enum exists so future triggers — time-based, idle — slot in).
  - **Risk formula**: `chance = clamp(baseChancePercent + Σ modifier contributions, 0, 100)`; each modifier reads a stat via the **quest condition namespace** (`needs.<id>`, `skills.<id>` — reuse `game/quests.ts`'s path resolution, don't reinvent) and contributes a linear interpolation from `pctAt0` (stat at 0) to `pctAtMax` (stat at its max). Example: kitchen fire `baseChancePercent: 2`, modifier `{ var: "skills.cooking", pctAt0: +15, pctAtMax: −2 }` — a novice cook risks ~17%, a master ~0%.
  - **Placement**: `"on"` spawns the accident on the base asset's own cells (kitchen fire ON the stove); `"adjacent"` picks a random free, in-bounds, non-wall cell 1–2 grid squares away (`adjacentRange`) — the designer's "leak" case. If no free cell exists, fall back to "on".
- **Runtime**: accident *instances* are runtime state (spawned into the world + nav/tap systems like placed objects), NOT written into the map file; shape must be serializable for the future save system (same convention as `QuestRunner.serialize()`). 
- **Interaction hierarchy (the designer's exact rule)**: if an accident instance's footprint **overlaps** a base asset's footprint, the accident's interactions REPLACE the base asset's in the tap menu (impossible to cook while the kitchen is on fire) and autonomy must likewise skip the blocked asset's actions. If they do **not** overlap, there is no hierarchy — the accident is just its own independent tappable object alongside the asset.
- **Tools**: the Asset Editor gains (a) nothing special for accident-category assets beyond the `clearedBy` action multi-select, and (b) an "Accidents" risk section on normal assets — accident dropdown (accident-category assets only), trigger, base %, placement + range, modifier rows with var dropdowns fed from `stats.json` (no free-typed ids, as always).

This is deliberate: several actions (e.g. "cook") ship `primaryNeed: null` and therefore never auto-stop on their own (main.ts's gain loop only auto-stops when a `primaryNeed` is set), so gating the roll on natural completion only would make their risk unreachable. `onActionStop` fires for every stop reason — natural auto-stop, player cancel, a fresh order overriding the current one — matching §7.3's "once, when a sim finishes an action" without over-specifying what "finishes" excludes. The SEPARATE no-duplicate-stacking rule (`AccidentRegistry.canSpawn`) is deliberately keyed by triggering instance (`THREE.Object3D.uuid` as `baseKey`, no schema change to `world.ts`'s placed-object `userData` needed) rather than geometry — a puddle that drifts under an unrelated chair still blocks that chair (geometric hierarchy), but that's independent from "don't let the SAME stove roll two simultaneous fires."

### 7.7 as-built (2026-07-14)

- **New module `game/marker.ts`**, split like doors.ts/accidents.ts/sprites.ts: pure logic (sparse config resolution, bob/spin math, mesh-kind classification, the synthetic `AssetDef` fed to `attachMesh`) is headless-tested in `test/marker.test.ts` with zero THREE dependency; the thin three.js layer (`createMarkerInstance`) turns that into a live group tracked every frame from `main.ts`'s render loop.
- **Anchor/tilt finding (verified by reading `sim.ts`, not guessed)**: `SimAgent.applyPose` only ever rotates the character ROOT (`object.rotation.x = -Math.PI/2`) for the pre-rig CAPSULE stand-in (`!agent.hasRig`) — the real rigged character never rotates its root for sit/lie, only `position.y` changes (seat/bed perch height); the seated/lying visual itself comes from the animation clip. So a literal `sim`-child marker would be safe from tilt in the common case — but `main.ts`'s `loadCharacter()` calls `sim.clear()` on every rig (re)load (first load, or a hot-reload `meshPath` edit), which would silently delete a marker parented there. **Decision**: the marker is an INDEPENDENT top-level object (a sibling of `sim` in the scene, not a child), and its `pivot.position` is copied from `sim.position` every frame — translation only, `sim`'s rotation is deliberately never read. This makes the marker immune by construction to any current or future root-rotation trick (the capsule tip hack, a facing snap, a future pose) rather than merely because today's specific cases were audited, and it sidesteps the `sim.clear()` deletion hazard entirely. Consequence: 3D spin needs no counter-rotation (the pivot never inherited a rotation to cancel) and sprites billboard for free (`THREE.Sprite` ignores rotation regardless).
- **Known limitation, documented not fixed**: the marker's vertical anchor is `sim.position.y + heightMeters + yOffset`, correct while standing/walking (`sim.position.y` is 0). While sitting/lying, `sim.position.y` shifts to `sitHeight`/`lieHeight` (seat-surface alignment, not the seated character's true head height), so `heightMeters` no longer represents the seated/lying sim's real head height and the marker reads as floating too high above a seated/lying sim. Precise pose-aware anchoring would need bone/socket lookups — out of scope for this slice, same "documented, future refinement" precedent as this repo's other approximate transforms (e.g. §8's HUD-collision accepted tradeoff).
- **Default mesh**: `THREE.OctahedronGeometry` (radius = half of a synthetic 1m "footprint" every marker mesh kind is fit to, for consistent `scale` behavior), Y-elongated ×1.4 for a plumbob silhouette, flat green (`0x3ec93e`) `MeshBasicMaterial` — deliberately basic/unlit (not `MeshStandardMaterial`) so the marker reads the same bright green at night as in daylight, matching a Sims plumbob's always-visible role rather than dimming with `applyDayNight`'s lighting.
- **Mesh resolution reuses §7.5's pipeline exactly, no reimplementation**: a synthetic `AssetDef` (`markerAssetDef`, id `overhead-marker`, `footprint:[1,1]`, `sprite:{orientation:'billboard'}`) is handed to `world.ts`'s exported `attachMesh`, which already does image-vs-GLB extension detection (`classifyMeshPath`) and the "instant stand-in, async swap, keep the stand-in on any failure" dance — the default octahedron IS that stand-in, so a bad/missing mesh path just leaves it in place, exactly like every other mesh call site.
- **Scale convention (deliberate deviation from ordinary assets)**: `scale` is NOT baked into the loaded mesh via `meshFit` (which only reapplies on a reload) — it's applied continuously to the outer wrapper group every frame instead, so `scale`/`yOffset`/`spinDegPerSec`/`bobAmplitude`/`bobHz` all stay instantly live-tunable for every mesh kind (including the async GLB/sprite ones) with zero rebuild. Only the `mesh` PATH triggers an async rebuild (`meshSig` check inside `update()`) — exactly the spec's "recreate marker on marker.mesh change, re-tune numbers in place otherwise" rule, verified live (see below).
- **Spin**: applies to both 3D kinds (the default octahedron AND a loaded `.glb`) — normalized world-yaw degrees via `spinAngleDeg`, set directly on `visual.rotation.y` each frame (no counter-rotation needed, see the anchor finding above); explicitly zeroed for the `'image'` kind (`THREE.Sprite` always faces the camera regardless of rotation — spinning it would be a no-op at best).
- **Sim-time wiring**: `main.ts`'s render loop calls `marker.update(sdt, data.tuning.character)` right after `anim?.update(sdt)`, using the same speed/pause-scaled `sdt` the mixer/doors/sprites already use — pause freezes spin/bob/GIF frames, 2×/3× speeds them up. GIF frame advance reuses the exact §7.5 mechanism (`visual.traverse((o) => o.userData.spriteUpdate?.(dtSeconds))`), since the marker isn't under `world` (so main.ts's existing `world.traverse` sweep doesn't reach it) and needs its own one-line sweep.

### 7.5 as-built (2026-07-14)

- **New module `game/sprites.ts`**, split like doors.ts/accidents.ts: pure logic (extension classification, sparse `sprite` config resolution, GIF frame-timing math, footprint-based plane sizing) is headless-tested in `test/sprites.test.ts` with zero THREE/browser dependency; a thin three.js layer (`createSpriteInstance`) turns an image path into a live visual.
- **Extension detection lives in ONE place**: `classifyMeshPath(path)` (`.glb`/`.gltf`/unrecognized/extensionless → `'model'`; `.png`/`.jpg`/`.jpeg`/`.webp`/`.gif` case-insensitively, query-string-tolerant → `'image'`) is called from a single shared function, `game/world.ts`'s now-**exported** `attachMesh(group, def, opts?)` — previously a private GLB-only helper. This is the ONE mesh-loading call site every consumer now goes through: `world.ts`'s `buildWorld()` (furniture, unchanged call), `doors.ts`'s `createDoorInstance` (door panels), and `accidents.ts`'s `buildGroup` (accident instances) — the latter two previously had their own near-duplicate copies of the GLB-load-then-swap logic, which this slice deleted in favor of the one shared function. Furniture and accidents get sprite support "for free" as a result; nothing else needed to change in either module besides swapping their inline loader for a call to `attachMesh`.
- **Door panels explicitly REJECT sprites** (the spec's "your call, document it"): `attachMesh(panel, def, { allowSprite: false })` from `doors.ts`. Rationale: a billboard sprite always faces the camera regardless of the hinge pivot's rotation (so it would never visually swing), and a floor-flat plane doesn't read as a vertical door panel at all — neither sprite orientation can represent a swinging door, so a door asset pointed at an image keeps its GLB-only stand-in box and logs a console warning rather than silently rendering something wrong.
- **Billboard mechanism (documented choice)**: `orientation: "billboard"` (the default) uses a real `THREE.Sprite` — three.js's built-in always-faces-the-camera primitive (full billboarding, not a manual Y-axis-only spin) — with `sprite.center.set(0.5, 0)` so the anchor sits at the sprite's bottom-center, matching how a GLB is grounded at y=0 by `normalizeModelToFootprint`. `orientation: "flat"` is an ordinary `PlaneGeometry` rotated flat (`rotateX(-Math.PI/2)`) with a `FLAT_EPSILON` (0.005m) lift off the floor to avoid z-fighting — a real mesh, so it reads correctly from any camera angle (including top-down), unlike a sprite. Both use `MeshBasicMaterial`/`SpriteMaterial` with `transparent: true, alphaTest: 0.5` (flat additionally `depthWrite: false, side: THREE.DoubleSide`) for clean alpha edges with no z-sorting artifacts.
- **Sizing**: `spritePlaneSize(def)` = `footprint` scaled by `meshFit.scale` (uniform number or per-axis `[x,y,z]`, z ignored — no depth on a 2D plane) — the same "meshFit is a deviation from the automatic fit" convention as GLBs (§7.1/§7.2). `meshFit.yOffset` nudges the plane/sprite vertically post-grounding, same field, same meaning, as a GLB.
- **GIF decode path**: prefers the browser `ImageDecoder` (WebCodecs) API — `decodeGifFrames()` fetches the file, decodes every frame up front into small canvases plus that frame's delay (`VideoFrame.duration` is in **microseconds**, converted to ms; a codec reporting no duration falls back to 100ms/frame), then during playback draws the current frame into a single shared `THREE.CanvasTexture`'s backing canvas and flips `needsUpdate` only when the computed frame index actually changes (not every render tick) — "keep the per-frame cost cheap" per the spec. Where `ImageDecoder` doesn't exist, or decoding throws, falls back to a plain `THREE.TextureLoader` load (the same path every `.png`/`.jpg`/`.webp` always takes) — for an animated GIF this naturally renders as a static first frame, since the browser's own `<img>`-style decoding used internally by `TextureLoader` doesn't animate.
- **Sim-time wiring**: `SpriteInstance.update(dtSeconds)` accumulates `dtSeconds*1000` into an internal ms counter and looks up the frame via `frameIndexAtTime` — called from `game/main.ts`'s render loop with `sdt` (already pause/1×/2×/3×-scaled, the exact same value the animation mixer and doors receive), via **one line**: `world.traverse((o) => { o.userData.spriteUpdate?.(sdt); })`. `attachMesh` stores the update callback on `group.userData.spriteUpdate` once the sprite is ready — since `world.traverse` recursively visits every descendant (furniture groups, `doors.group` and accident groups are both parented under `world`), this single traversal advances GIF frames anywhere in the scene with zero additional per-caller wiring. No new signature changes were needed on `buildWorld`, `buildDoors`, or `AccidentsController` to make this work.

  - `/data/*.json` and `/api/*` → **network-only**, the fetch handler never touches the cache for these at all (`isDataOrApiPath()` in `sw-routing.js`, headless-tested in `test/sw-routing.test.mjs`). The SW also only ever intercepts `GET` requests (`if (req.method !== 'GET') return;`), so tool `PUT`/`DELETE` calls are never even looked at by the fetch handler, belt-and-suspenders on top of the path check.
  - Everything else (app shell, tool pages, built JS/CSS, models, manifest, icons) → **network-first**, falling back to a versioned cache (`condo-life-v1`) only when the network request fails (offline); successful responses are opportunistically cached for that fallback. Chosen over cache-first specifically because `npm run dev` serves `game/*.ts` through Vite's own dev transform (HMR, query-stamped URLs) — a cache-first policy would risk serving a stale dev module. Network-first means the cache is simply never consulted while the network is up, in dev or prod, so it can't interfere with HMR or a tool's live-edit workflow, while still giving real offline support once installed.
  - **Verified live** (`npm run dev` + browser, SW active): `navigator.serviceWorker.getRegistrations()` showed the SW `activated`; a direct `fetch('/data/tuning.json', {cache:'no-store'})` still returned the live file (200, `cache-control: no-store` from `server.js`); the game's own 2s hot-reload poll continued returning fresh 200s for all `/data/*.json` files with the SW active (checked via the browser's network log); a tool page (`/tools/assets.html`) loaded with zero console errors and no SW registered on it (registration script isn't present there).

### 7.8 Sit/lie perch fix + designer-editable use-position (roadmap item 1, 2026-07-15)

- **Bug**: characters sat/lay completely outside furniture (bed: lying beyond/across it; sofa/armchair/dining chair: sitting on the floor beside it).
- **Root cause (found by reading `game/sim.ts`'s `applyPose`, not guessed — see `game/facing.ts`'s `usePoseFor` doc comment for the numeric walkthrough)**: two independent bugs, both in `applyPose`, meant the perch-snap code path essentially never ran:
  1. It compared `action.animation` against the EXACT strings `'sit'`/`'lie'`, but every shipped action's `animation` value is a clip name like `"sit_idle"`, `"sit_eat"`, `"lie_sleep"`, `"stand_use"` (`data/interactions.json`) — none of those equal `'sit'` or `'lie'` literally, so the comparison was false for every real action and the function returned immediately, doing nothing.
  2. Even fixing that, `perch` was computed as `a.seat ?? (anim === 'lie' ? a.target : null)` — a plain non-seat-aware "Sit" action (e.g. tapping an armchair's own "sit", not a seat-aware "Watch TV") has no resolved `seat` (only `seatAware` actions like `watch_tv` get one via `findSeatFor`), so `perch` was `null` and the function still no-op'd for the single most common case (sitting directly on the furniture you tapped).
  - Net effect: the sim was simply left standing at its walk-up APPROACH point (`useSpotFor`'s point just outside the footprint edge, by design for walking towards an asset) — never repositioned onto the seat/bed at all. Not a pivot-math or footprint-center bug; the position math was fine, it just never executed.
- **Fix**: `applyPose` now derives the pose category from the animation string's PREFIX (`startsWith('lie')` / `startsWith('sit')`, covering every shipped value), and `perch` defaults to `a.seat ?? a.target` (so a direct sit/lie on the tapped object itself is covered, not just seat-aware target→seat cases). Stand actions (`stand_use`, prefix neither) are left alone — they already end up standing at a sensible in-front spot via `useSpotFor`, nothing to snap.
- **Schema — `AssetDef.usePose?: { sit?: UsePoseEntry; lie?: UsePoseEntry }`** (`game/data.ts`), `UsePoseEntry = { offset?: [number, number]; y?: number; facingDeg?: number }`, fully sparse (any/all fields absent). New pure function `game/facing.ts`'s `usePoseFor(pose, instance, def, tuning)` computes the perch transform:
  - **Default (no `usePose` at all)**: position = the footprint CENTER (`instance.pos`, i.e. `offset [0,0]`) instead of the old approach point; height = `tuning.character.sitHeight`/`lieHeight` (unchanged fallback constants, 0.25/0.55); facing = `worldFacingDeg(instance, def)` — for a bed this is the SAME direction as its long axis (footprint depth is local Z, the axis `facingVector`'s "local +Z is forward" convention treats as forward), so a sim lying down with zero configuration already aligns with the bed's long axis, fixing the reported "lie orientation" bug as a side effect of the default, not a special case.
  - `offset` is MODEL-LOCAL, rotated by the placed instance's `rotDeg` (`rotateLocalOffset`, the same rotation-matrix convention as `facingVector`) — a placement nudge, not a direction, so it composes with instance rotation the way a raw furniture offset should (NOT via `worldFacingDeg`, which is for directions like `facingDeg`).
  - `facingDeg` in an entry is model-local like `AssetDef.facingDeg` itself: world facing = `instance.rotDeg + entry.facingDeg`.
  - A seat-aware action's subsequent "face the target" step (already existing in `sim.ts`'s `update()`, e.g. facing the TV from the sofa) still runs AFTER `applyPose` and overrides the returned `facingDeg` when target ≠ perch — `usePoseFor` only supplies the perch's own default/overridden facing, not the final in-scene rotation.

## 7.30 Behavior/autonomy system — locked design (B8-1)

- **data/behavior.json**: `{ weights: { needDeficit, distance, personalityAffinity }, rules: [{ id, name, action?: actionId, assetCategory?/assetId?, conditions?: Condition (quest namespace incl. personality.*), scoreBonus: number, enabled }], needWeights?: { <needId>: number } }`. 
- **Engine (game/behavior.ts, pure)**: replaces/extends Autonomy's "lowest need" pick with utility scoring over every candidate (placed asset × its autonomy-eligible actions): `score = Σ needDeficit(need)*needWeights[need]*assetGainRate(need) − distancePenalty + Σ matching rule scoreBonus (conditions evaluated via quests.ts resolveVar)`. Highest score wins if above a threshold (tunable). Asset differentiation falls out of per-asset gain rates (sofa nap low energy gain vs bed sleep high). Personality enters via rule conditions/bonuses (e.g. low cleanliness ⇒ negative bonus on clean_up). Backward compat: absent behavior.json ⇒ legacy lowest-need behavior.
- **Tool (tools/behavior.html)**: weights editors, rule CRUD (condition builder reused from Quest Editor pattern), and a LIVE SCORE PREVIEW: table of current candidates with their computed scores against the live game state (fetch live? no — computed against editable hypothetical stat values in the tool). nav.js entry, jsdom suite.

## 7.32 Floor + wall textures — as-built (B9-1)

- **Schema**: `MapData.floors[].texture?` and `MapData.walls[].texture?` (image path under `public/`, e.g. `"textures/oak.jpg"` — `normalizeMeshUrl` adds the leading slash; same drop-in convention as `AssetDef.mesh`). New global tunable `tuning.textures.metersPerTile` (default 1) = how many meters of surface one texture repeat spans, so tiling reads at a physical scale independent of room size. All optional/sparse; absent = today's flat `FLOOR_COLORS`/wall color, zero change to existing maps.
- **Pure math (`game/textures.ts`, headless-tested `test/textures.test.ts`)**: `resolveMetersPerTile(tuning)` (guards absent/0/negative/NaN → 1m), `textureRepeat(surfaceMeters, metersPerTile)` = `surfaceMeters / metersPerTile` (the tile count across that span; guards → 1m tile / 0 repeat), `polygonBounds(polygon)` = axis-aligned floor bounds for UV normalization + per-axis sizing.
- **Repeat math**: a surface with **0..1 UVs** across a dimension gets `repeat = spanMeters / metersPerTile`. Walls use BoxGeometry face UVs directly (`repeat = [len/mpt, WALL_H/mpt]`, WALL_H=2.5). Floors: ShapeGeometry emits UVs in raw shape-space **meters**, so `normalizeFloorUVs` remaps them to 0..1 over `polygonBounds`, then `repeat = [w/mpt, h/mpt]` — giving both surface types identical physical tiling.
- **Engine (`game/world.ts`)**: `loadTexture(url)` caches the decoded image per URL (mirrors `gltfCache`); each surface **clones** the cached texture (`.repeat`/`.wrapS` live on the Texture, so different-length walls/different-size floors each need their own) — clone shares `.image`, cheap. `applySurfaceTexture` is keep-stand-in: the mesh renders its color material immediately, swaps in the tiled texture (RepeatWrapping, SRGB colorSpace, color→white so the image shows) on load success, keeps the color on any load failure. Textured floors/walls each get their **own** material instance (walls otherwise share one). Hot-reload re-resolves for free (module-level cache survives `buildWorld` rebuilds).
- **Wall-cut view**: unchanged — it only scales wall geometry `.scale.y`. A textured wall's v-repeat is computed from the FULL height, so during a cut the texture simply compresses vertically with the geometry (accepted, not re-mapped).
- **Tool + server (DONE 2026-07-16):** `GET /api/textures` (server.js) flat-lists `public/textures/*.{png,jpg,jpeg,webp}` as `'textures/<file>'` paths (missing/empty dir → `[]`, read-only). Map Editor Floors + Walls inspectors have a shared `textureRow` picker ('(none)' = color, else a listed path → `floors[]`/`walls[].texture`) with a live preview swatch; textures fetched once at boot into `state.textures`. Designer drops images in `public/textures/`, picks per surface, saves — the engine load path above then renders them.
- **Per-surface texture scale (follow-up, DONE 2026-07-16):** optional `floors[].textureScale?`/`walls[].textureScale?` (default 1) multiplies that one surface's `metersPerTile` via `effectiveMetersPerTile(metersPerTile, scale)` in `game/textures.ts` (guards non-finite/non-positive → 1x); `world.ts` computes the effective value per floor/wall before calling `textureRepeat`. `textureRow` in `tools/map.html` grew a sparse number input next to the texture dropdown, shown only while a texture is selected; typing 1 or clearing it deletes the key. Tested in `test/textures.test.ts` and `test/map-editor.test.mjs`.
- **Per-side wall textures (follow-up, DONE 2026-07-16):** optional `walls[].textureB?` — when set, the wall's two big faces differ (`texture` = side A, `textureB` = side B); absent = both faces use `texture`, unchanged behavior. **A/B convention (geometric, computed at render time from the wall's actual placement — independent of which endpoint is `from` vs `to`):** a wall running mostly along X ("horizontal") has its two big faces pointing world +Z/-Z — side A is whichever faces +Z ("south"); a wall running mostly along Z ("vertical") has faces pointing +X/-X — side A faces +X ("east"). Documented in `game/data.ts` (walls doc) and `world.ts`. Engine: `world.ts`'s wall loop only builds a 6-entry `THREE.Material[]` (BoxGeometry's default face-group order `[+x,-x,+y,-y,+z,-z]`, so the two big faces are indices 4/5 = local ±z) when `textureB` is present; otherwise the existing single-material path is untouched. `applySurfaceTexture` was refactored to take a `THREE.MeshLambertMaterial` directly (not the mesh) so it can target either the sole wall material or one of the two per-side materials; floor call site updated to match, no behavior change there. Edge faces (0-3, the wall's tiny end-caps/top/bottom) reuse side A's material. `textureScale` applies to both sides equally. Tool: Walls inspector's existing dropdown relabeled "Texture (side A / both)"; new sparse "Texture side B (optional)" dropdown + swatch (`textureSideBRow` in `tools/map.html`) with the A/B orientation convention in its title hint. No new pure-math helper — this is three.js material wiring, not sizing math. Tested in `test/map-editor.test.mjs` (side-B round-trip: renders defaulting to "(same as side A)", offers listed textures, sets/clears `walls[].textureB` sparsely, swatch reflects selection).
- **Black wall tops (B10-1 follow-up, DONE 2026-07-16):** every wall now ALWAYS gets a 6-entry material array (the `textureB`-only condition is gone) whose index 2 (+y, top face) is one shared unlit `THREE.MeshBasicMaterial` — flat, unshaded, untextured — for the designer's "architecture plan" look in the wall-cut view. Color = `tuning.view.wallTopColor` (sparse, default `'#000000'`; typed in `game/data.ts` `view?`, listed in `data/tuning.json`, auto-surfaced by the generic Tuning Editor). Bottom (index 3) and end-cap faces stay side A; the 4/5 side-A/B logic is unchanged, just expressed against the always-array structure. `applyWallCutView` needed no change (it only scales geometry Y).

## 7.55 Generalized asset states (New.txt #3) — as-built (2026-07-20)
