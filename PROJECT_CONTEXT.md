## 2. Character & animation system (as built)

- `game/anim.ts` ‚Äî `AnimController` over a `THREE.AnimationMixer`. Logical states (`idle`, `walk`, `sit`, `lie`, or any action's `animation` field value) resolve to clip names through `tuning.character.clipMap` (exact ‚Üí case-insensitive ‚Üí substring). Missing clips warn once and fall back to idle. Cross-fade duration, walk-clip playback rate (`walkSpeed / walkClipSpeedReference`), sit/lie perch heights ‚Äî all in `tuning.character`.
- `game/world.ts` ‚Äî `loadRiggedCharacter()` loads `tuning.character.meshPath`, **merges clips from every GLB in `tuning.character.animationPaths`** (Mixamo-style one-clip-per-file exports; same skeleton required), normalizes to `heightMeters` (centered, grounded), optional `yawOffsetDeg` wrapper for models not facing +Z. Load failure keeps the capsule stand-in.
- `game/sim.ts` ‚Äî `onLocomotionChange` callback drives idle‚Üîwalk; `hasRig` flag disables the capsule-era tip-over lie hack; perch heights from tuning.
- `game/main.ts` ‚Äî mixer advances on **sim time** (pause freezes, 2√ó/3√ó speeds up). Hot-reload re-tunes the controller and re-loads the rig when `meshPath` or `animationPaths` change.
- **`tools/animations.html` ‚Äî Animation Mapper**: lists the model's actual clips, previews any clip on the character in 3D, maps clips to states via dropdowns, manages animation source GLBs, and edits all rig numbers. States shown = core (`idle/walk/sit/lie`) + every `animation` value referenced by actions + designer-added custom states. Unmapped states flagged ‚öÝ. Saves the whole `tuning.json` (character block edited, other groups preserved).
- **T-pose diagnosis**: T-pose = no clip resolved. Open the Animation Mapper ‚Äî if the clip list is empty, the animations are in separate files: add them under *Animation sources*. Otherwise the clipMap names just don't match: remap from the listed clips.
- The user's desired state vocabulary (idle, walking, running, sitting, laughing, talking, screaming, cooking, sleeping, reading‚Ä¶): `idle`/`walk` are automatic; everything else becomes live when an **action**'s `animation` field references it. "Running" would need a game-side locomotion feature (e.g. run-when-far) ‚Äî not built; treat as a future tuning flag.

## 2b. Maps & the active-map mechanism (as built, Map Editor slice 2)

- **`tools/map.html`** is now the complete Map Editor (slices 1+2 in one file ‚Äî note: the user's tools folder was missing the slice-1 map.html, so this file is standalone and authoritative). Modes: **Objects** (palette add, drag with ¬Ω-cell snap, R rotate with 90¬∞ normalization, inspector x/z/rot), **Floors** (drag to draw rectangle floors with a chosen material ‚Äî compose several rects for L-shaped rooms; click to edit id/material/delete), **Walls** (drag to draw, axis-locked and snapped; click to edit endpoints numerically or delete), **Doors** (click a wall to place ‚Äî position projects onto the wall and snaps along it, orientation inferred from wall axis; drag to move, R flips orientation), **Spawn** (click to place, facing¬∞ in inspector). Plus: **Ctrl+Z undo** (50-step snapshot stack, closes a known gap), floor-plan **underlay** image with opacity/width/offset controls (session-only tracing guide, not saved), nav overlay (game's own `nav.ts` via Vite, rebaked lazily on mutation), map-properties editing (name, gridSize, bounds).
- **Multiple maps**: header select + **New / Duplicate / Delete** (new & duplicated maps are PUT to `data/maps/<id>.json` immediately). `server.js` gained `GET /api/maps` (listing) and `DELETE /api/data/maps/*.json` (deletion is refused for non-map files). Deleting the active map or the last map is blocked in the tool.
- **Active map**: `tuning.map.active` names which `data/maps/<id>.json` the game plays (default `condo`). `data.ts` loads tuning first, then that map. The editor's **"‚ñ∂ Play this map"** button PUTs tuning with the current map id ‚Äî the running game hot-reloads into it, and `main.ts` detects the map-id change and teleports the sim to the new spawn (`SimAgent.teleportTo`).

## 2c. Needs & skills add/remove (as built, Tuning Editor extension, 2026-07-13)

- **No new tool file** ‚Äî `tools/tuning.html`'s existing Needs/Skills sections already render one editable group per entry (`renderNeeds`/`renderSkills`), so add/remove was a natural extension rather than a new Stats Editor page. A dedicated `tools/stats.html` would have duplicated that machinery for no benefit; extending was the cleaner call here (contrast with the Map Editor, which got its own file because it needed a canvas + very different interaction model).
- **Add**: "+ Add need" / "+ Add skill" toolbar buttons (`[data-action="add-need"|"add-skill"]`) at the top of each section. `window.prompt` asks for a display **name only**; the id is derived (`slugify` ‚Üí lowercase, non-alphanumerics ‚Üí `_`) and uniquified against the existing list (`thirst`, `thirst_2`, ‚Ä¶) ‚Äî no separate id prompt, no failure path to retry. New-entry defaults are **averaged from existing (non-computed) entries**, not hardcoded: a need copies the mean `default`/`decayPerTick` of current needs and starts `autonomy: true`; a skill starts at `default: 0` (skills grow through practice, never decay ‚Äî no `decayPerTick` field exists on `SkillDef`) with `max` averaged from existing skills. Bar `color` is randomized (designer edits it via the existing color field same as any entry).
- **Remove**: per-entry "Delete need"/"Delete skill" buttons (`[data-action="delete-need"|"delete-skill"][data-need-id|data-skill-id]`). Referential integrity is checked against `interactions.json`: a need can be referenced by an action's `needGains` key or its `primaryNeed`; a skill only by `skillGains`. **Chosen UX (deliberately different from the Asset Editor's "warn and leave dangling" pattern)**: the confirm dialog lists every referencing action by name and field, and on confirm the tool **strips the dangling references** (deletes the `needGains`/`skillGains` key, nulls a matching `primaryNeed`) in the same save cycle ‚Äî `interactions.json` is marked dirty alongside `stats.json` so both PUT together. Rationale: `game/stats.ts`/`game/autonomy.ts` already no-op safely on an unknown id (nothing crashes either way), but shipping a `needGains`/`skillGains` key that points at nothing is exactly the kind of orphaned data this data-driven design is supposed to avoid, and stripping is a single confirm click away from being wrong-footed by a docs/CLAUDE.md-style comment describing a field that no longer means anything.
- **Ordering gotcha (documented in code comments)**: `render()` fully rebuilds section DOM (`innerHTML = ''` + re-append), which discards any `.dirty` class applied beforehand ‚Äî `markDirty(file)` must be called **after** `render()` in the add/remove handlers, not before, or the dirty indicator (and the Save button's enabled state) silently fails to reflect the change. (Plain field edits don't hit this because they mutate in place without re-rendering.)
- **No hardcoded ids found in game code**: audited `game/*.ts` for literal need/skill strings (`hunger`, `energy`, `english`, etc.) ‚Äî none exist. `game/stats.ts` (`SimStats`), `game/ui.ts` (`Hud.rebuildBars`), and `game/autonomy.ts` all iterate `stats.needDefs`/`skillDefs`/`lowestAutonomyNeed()` purely by whatever ids are in `stats.json`; the Environment "computed" need is found dynamically (`data.stats.needs.find(n => n.computed)` in `game/main.ts`), not by id. Confirmed live: added/removed needs and skills render on the in-game HUD and drive decay/autonomy with zero code changes.
- **Tests**: `tools/tuning-editor.test.mjs` extended (not a new suite) with add/cancel/uniquify coverage for both needs and skills, and both referential-integrity branches (unreferenced ‚Üí plain message; referenced ‚Üí lists the action(s), strips the reference on confirm, cancel leaves everything untouched), plus a final PUT assertion proving `stats.json` and `interactions.json` save the fully-reconciled state together.
- **Environment note**: this session also discovered the repo's `node_modules` had never actually been installed (no `jsdom`/`tsx`/`canvas`) ‚Äî every existing test suite would have failed to even start. Added `jsdom`, `tsx`, and `canvas` as devDependencies so the full suite (tool `.mjs` + game `.ts`) is runnable; this is a one-time environment fix, not a code change.

## 2d. Interaction Editor (as built, 2026-07-13)

- **New tool file `tools/interactions.html`** ‚Äî sidebar (all actions, "player-only" badge when `autonomyEligible` is false) + editor panel, following the Asset Editor's layout (contrast with ¬ß2c's needs/skills, which extended an existing section instead ‚Äî actions have enough fields and their own referential-integrity direction that a dedicated page was the cleaner call here, same reasoning as why Map Editor got its own file).
- **Autonomy-eligibility flag: pre-existing, not added.** `data/interactions.json`'s `ActionDef` already carried `autonomyEligible: boolean` on every action (`game/data.ts` `ActionDef` interface, line ~13), and `game/autonomy.ts`'s `maybeAct()` already gates candidates on it (`if (!action || !action.autonomyEligible) continue;`) before matching `primaryNeed`. No schema change and no `autonomy.ts` change were needed ‚Äî the editor just exposes the existing flag as a checkbox, so e.g. "Wash hands" or "Cook" (already `autonomyEligible: false` in the shipped data) stay player-only, and toggling it live-round-trips through the sim's autonomy search on save.
- **Editable per action**: name, free-text `animation` state (datalist + hint line listing every state already in use ‚Äî the 4 core states `idle`/`walk`/`sit`/`lie` the Animation Mapper always recognizes, plus every other action's current `animation` value ‚Äî so the designer isn't guessing a name that already drives a mapped clip), `autonomyEligible`, `seatAware` (sparse ‚Äî key only present when true, mirroring the Asset Editor's `seatTarget` convention), `primaryNeed` (dropdown fed from `stats.json` needs + "(none)", writes `null` not a missing key ‚Äî matches the schema's `primaryNeed: string | null`), and one numeric row per need/skill in `stats.json` for `needGains`/`skillGains` (blank = no entry in the map) ‚Äî same "fed from data, no free-typed ids" pattern as the Tuning Editor's per-action gain rows, just given a dedicated full-field page instead of a compact grid.
- **id is fixed** once created (assets reference it by string, like an asset's own id) ‚Äî editing swaps the display name only.
- **Add**: "+ New action" prompts for a display name only; id = `slugify(name)` uniquified against existing action ids (`do_yoga`, `do_yoga_2`, ‚Ä¶) ‚Äî identical derivation helper to the Tuning Editor's `addNeed`/`addSkill`. Defaults: `needGains: {}`, `skillGains: {}`, `animation: ''`, `autonomyEligible: true`, `primaryNeed: null` (sane no-ops until the designer fills them in).
- **Referential integrity** is checked against `data/assets.json` (an asset's `interactions: string[]` array). Delete follows the same "warn with names, strip on confirm" policy as ¬ß2c: unreferenced action ‚Üí plain confirm; referenced action ‚Üí confirm dialog lists every referencing asset by name, and on confirm the tool removes the id from those assets' `interactions` arrays and deletes the action, marking **both** `interactions.json` and `assets.json` dirty so they save together in one PUT cycle ‚Äî `assets.json` never ships pointing at an action id that no longer exists.
- **Hot-reload**: works with zero changes. `game/data.ts`'s `watchData()` already polls and reloads the whole `GameData` bundle (including `interactions.json`) every 2s via `loadAll()`, and `game/main.ts`'s hot-reload callback does `data = fresh` (a `let`, not `const`) ‚Äî every consumer that reads `data.interactions.actions` at use-time (e.g. building the action menu, `WatchTick`-equivalent gain application, `autonomy.ts`'s `actionsById` map) picks up edited/added/removed actions on the next poll with no extra plumbing.
- **Tests**: new suite `test/interaction-editor.test.mjs` (placed alongside the other newest suites, `test/map-editor.test.mjs` and `test/animation-editor.test.mjs`, rather than `tools/` where the two oldest suites still live) covers render + player-only badge, editing every field including the sparse `seatAware` key and the `primaryNeed` null round-trip, add with slugify+uniquify, both delete branches (unreferenced plain-confirm; referenced confirm-lists-assets-and-strips, asserting the exact stripped `assets.json` PUT payload), and search filtering.

## 3. Quest system ‚Äî locked design (build in a future slice)

Replaces the roadmap's vaguer "Story Editor". Two parts:

### 3.1 Sim-state variables (`data/simstate.json` or a `variables` block)
Designer-defined variables beyond needs/skills, e.g. `visaStatus` (string), `job` (string|null), `income` (number), plus the built-in `funds`. Defaults live in data; the save system persists current values. Variables are readable by quest conditions and writable by quest rewards. This also finally gives `tuning.economy.startingFunds` a consumer.

### 3.2 Quests (`data/quests.json`) ‚Äî draft schema
```json
{
  "quests": [{
    "id": "first_words",
    "name": "First Words",
    "description": "Reach English level 10 and save ¬ß500.",
    "trigger":    { "all": [ { "var": "skills.english", "gte": 1 } ] },
    "completion": { "all": [ { "var": "skills.english", "gte": 10 },
                             { "var": "funds", "gte": 500 } ] },
    "rewards": [ { "type": "funds", "amount": 200 },
                 { "type": "setVar", "var": "visaStatus", "value": "student" } ],
    "onceOnly": true
  }]
}
```
- **Condition namespace**: `needs.<id>`, `skills.<id>`, `funds`, `time.hour`, `time.day`, `vars.<name>`, `quests.<id>.state` (`locked|active|done`).
- **Operators**: `gte`, `lte`, `eq`, `neq`; combinators `all` / `any`, nestable.
- **Reward types**: `funds`, `setVar`, `unlockAsset` (hides/shows catalog entries ‚Äî ties into Buy/Sell mode), later `dialogue` / vocabulary popups for the language-learning content.
- **Evaluation**: on the needs-decay tick (same reuse-an-existing-interval convention as autonomy). Quest state persists in the save.
- **Quest Editor tool**: CRUD quests, condition builder with dropdowns fed by `stats.json`/variables (no free-typed ids), live validation, and a quest log HUD panel in-game (trigger ‚Üí "quest started" toast; completion ‚Üí rewards applied + toast).

Build order within the quest slice: 1) variables + condition evaluator (headless-tested ‚Äî this is pure logic, perfect for the test suite), 2) in-game quest runner + HUD log, 3) Quest Editor tool.

## 3.3 Quest system runtime ‚Äî as built (steps 1‚Äì2, 2026-07-13)

Steps 1‚Äì2 of ¬ß3's build order are shipped (game side only ‚Äî no Quest Editor tool yet, that's step 3).

- **New data files**: `data/simstate.json` (designer variable defaults ‚Äî `visaStatus` string, `job` string|null, `income` number, each `{ id, name, type, default }`; `funds` is deliberately NOT in this file, see below) and `data/quests.json` (shipping the exact `first_words` example from ¬ß3.2, ids matched to the real `stats.json`: `skills.english` ‚Äî English's `max` is 10, matching the quest's `gte: 10` completion threshold). Both are generic `data/*.json` files; `server.js`'s `/api/data/<path>` and `/data/<path>` routes already serve/accept any `.json` under `data/` with no code change needed ‚Äî verified by loading both through the existing `GET/PUT` handlers.
- **New module `game/quests.ts`** (pure, no DOM/three.js ‚Äî headless-testable):
  - `evaluate(condition, ctx)` ‚Äî recursive evaluator for the ¬ß3.2 condition tree (`all`/`any`, `gte`/`lte`/`eq`/`neq`) against an `EvalContext` snapshot (`needs`, `skills`, `funds`, `time.{hour,day}`, `vars`, `quests` ‚Äî all plain records, built fresh each tick by the caller).
  - **Unknown-id semantics (decision)**: any `var` path that doesn't resolve (unknown need/skill/var/quest id, or an unrecognized namespace) resolves to `undefined`, and every operator against `undefined` is `false` ‚Äî never throws. A leaf with none of `gte/lte/eq/neq` set is likewise `false`. `all([])` is vacuously `true`; `any([])` is `false` (nothing to satisfy) ‚Äî both just `Array.prototype.every`/`some`'s native behavior, called out explicitly in the code comment as intentional.
  - `QuestRunner` ‚Äî the runtime state machine: `quests: Record<id, 'locked'|'active'|'done'>`, `vars: Record<id, value>`, `funds: number`, `unlockedAssets: Set<string>`, `completedLog: string[]`. `tick(needs, skills, time)` evaluates all `locked` quests' triggers (‚Üí `active`, fires `onQuestStarted`), then all `active` quests' completions (applies rewards, fires `onQuestCompleted`) in the same pass ‚Äî a quest can in principle trigger and complete in the same tick if its completion condition already holds; documented as intentional rather than special-cased, since ¬ß3.2 doesn't require a one-tick delay.
  - **onceOnly semantics**: `onceOnly: true` quests go `active ‚Üí done` and stay `done` forever (rewards apply exactly once). `onceOnly: false` quests go `active ‚Üí done` (rewards apply) then **immediately reset to `locked`**, so they can trigger again later ‚Äî this is the "repeatable quest" reading of ¬ß3.2's "onceOnly:true quests never re-trigger" (which implies the `false` case can). Not explicitly speced beyond that phrase; flagged here as a design call for the Quest Editor agent to know about (a repeatable quest's rewards fire every time it completes).
  - **Rewards**: `funds` (add to balance), `setVar` (write `vars[name] = value`), `unlockAsset` (add an asset id to `unlockedAssets`).
  - **unlockAsset / Buy-Sell tie-in**: implemented as an additive `Set<string>` of asset ids a quest has explicitly unlocked (`isAssetUnlocked(id)` helper), not a schema change to `assets.json` ‚Äî no asset currently opts into being quest-gated. Documented expectation for the future Buy/Sell mode (roadmap ¬ß4 item 6): an asset is purchasable if it has no quest-gate flag OR `isAssetUnlocked` is true; that gate flag itself (e.g. a future `AssetDef.requiresQuestUnlock`) is for that slice to add.
  - **Hot-reload (`retune`)**: adopts new/edited quest and variable *definitions* without touching runtime *state* ‚Äî existing quest states and variable values are preserved; only brand-new quest ids get `locked` and brand-new variable ids get their default. Removed ids leave an orphaned (harmless) runtime entry, mirroring `stats.ts`'s `retune()` convention.
  - **Persistence ‚Äî deferred**: there is no save system in this repo (`data/save/` is empty and gitignored; no `save.ts` module exists anywhere). Quest/variable/funds state is therefore **runtime-only** (resets on page reload), same as needs/skills today. `QuestRunner.serialize()`/`restore(QuestSaveState)` produce/consume a flat JSON-serializable shape specifically so wiring a future save system is a direct `JSON.stringify(runner.serialize())` / `runner.restore(parsed)` call, no refactor needed.
- **`game/data.ts`** gained `VarDef`/`SimStateData`, `Condition`/`ConditionLeaf`/`ConditionAll`/`ConditionAny`, `QuestState`, `Reward`/`RewardFunds`/`RewardSetVar`/`RewardUnlockAsset`, `QuestDef`/`QuestsData` types; `GameData` gained `simstate`/`quests`; `TuningData` gained `quests: { toastDurationSeconds, completedLogLimit }` (both now in `data/tuning.json`, no hardcoded numbers); `loadAll()` fetches the two new files alongside the rest.
- **`game/main.ts`** wiring: `QuestRunner` constructed after `stats`/`hud`/`autonomy`, seeded with `data.tuning.economy.startingFunds` (giving that field its first real consumer, per ¬ß3.1). `quests.tick(...)` is called **inside the existing needs-decay `while` loop** in `simTick` (no new accumulator/timer ‚Äî reuses `decayAcc`/`needsDecayTickSeconds`, per ¬ß3.2's "same reuse-an-existing-interval convention"). Needs/skills are snapshotted per-tick via `Object.fromEntries(stats.needs/.skills)`. `time.day` is new: the render loop's clock advance changed from `gameSeconds = (gameSeconds + ‚Ä¶) % 86400` to an explicit `while (gameSeconds >= 86400) { gameSeconds -= 86400; gameDay++; }`, so `time.day` (1-indexed, starts at 1) is now tracked and available to conditions. Quest start/complete callbacks trigger HUD toasts and rebuild the quest log; hot-reload calls `quests.retune(...)` + rebuilds the log.
- **HUD (`game/ui.ts`)**: new collapsible `#quest-panel` (bottom-left, above the devbar, same visual language as the needs/skills panels) listing active quests (name + description) and a "Completed" section (last `tuning.quests.completedLogLimit` entries, newest first). New `#quest-toasts` stack (top-center, non-interactive/`pointer-events:none` so it never blocks taps) shows transient "Quest started: X" / "Quest completed: X" toasts, auto-dismissing after `tuning.quests.toastDurationSeconds`. `Hud.setQuestLog(active, completed, limit)` and `Hud.showQuestToast(text, kind, durationMs)` are the new public methods.
- **Hot-reload semantics (documented behavior)**: `quests.json`/`simstate.json` flow through the existing `watchData()`/`loadAll()` 2s poll exactly like every other data file ‚Äî editing a quest's name/description/conditions/rewards on disk live-updates the definition, but a quest already `active` or `done` stays that way (only its displayed name/description/conditions change); a brand-new quest in the file starts `locked` and evaluates normally from then on.
- **Tests**: new `test/quests.test.ts` (`npx tsx test/quests.test.ts`) ‚Äî operators (all 4), every namespace including `quests.<id>.state`, `all`/`any` nesting (incl. empty-array vacuous cases), unknown-id/malformed-leaf ‚Üí false for every namespace, full runner lifecycle (locked‚Üíactive‚Üídone, onceOnly true and false/repeatable, all 3 reward types, `retune` preserving state while adopting new defs, `serialize`/`restore` round-trip). All existing suites re-run clean: `anim.test.ts`, `charfit.test.ts`, `nav.test.ts`, `locomotion.smoke.ts` (via `npx tsx`), `animation-editor.test.mjs`/`interaction-editor.test.mjs`/`map-editor.test.mjs` (via `node`, under `test/`), `asset-editor.test.mjs`/`tuning-editor.test.mjs` (via `node`, under `tools/`). **Pre-existing quirk noted, not introduced by this slice**: `test/meshfit.test.mjs` fails under plain `node` (`ERR_MODULE_NOT_FOUND` ‚Äî it imports `../game/world` without an extension, which only Node's ESM-extensionless resolution under `tsx` handles) but passes under `npx tsx test/meshfit.test.mjs`; left as-is since it's unrelated to this slice. Strict type gate (`npx tsc --noEmit --strict --target es2020 --moduleResolution bundler --module esnext --skipLibCheck game/*.ts`) is clean.

## 3.4 Quest Editor tool (as built, 2026-07-13)

Step 3 of ¬ß3's build order ‚Äî the last piece of the quest slice. **New tool file `tools/quests.html`**, no game-module imports needed (pure JSON editing, so it's a plain inline script only, same as `tools/tuning.html` ‚Äî no `type="module"` section, contrast with the Map Editor / Animation Mapper which genuinely need `nav.ts`/`world.ts`).

- **Layout**: one page, two `<section>`s stacked (mirrors `tools/tuning.html`'s multi-section single-column layout, not `tools/interactions.html`'s single sidebar+editor, since this tool has two distinct entity kinds). **Variables** (`data/simstate.json`): a toolbar (`+ Add variable`) + one row per `VarDef` ‚Äî name, id (fixed after creation), a `type` dropdown (`string`/`number`/`boolean`, matching the real `VarDef.type` in `game/data.ts` ‚Äî not the draft `string|null` phrasing in the slice brief, which the shipped type superseded), and a default-value control with a separate "null" checkbox (needed because `VarDef.default` can be `string | number | boolean | null` independent of `type` ‚Äî e.g. the shipped `job` variable is `type: 'string'` with `default: null`). Changing `type` coerces any non-null default to a sane value of the new type. **Quests** (`data/quests.json`): a sidebar (name + a "repeatable" badge when `onceOnly` is false) + an editor panel, same visual pattern as `tools/interactions.html`.
- **Condition builder**: recursive renderer over the `Condition` tree (`isLeafCond`/`renderGroupBox`/`renderLeafRow`). Every rendered node carries a `data-cond-path` positional address (`"trigger"`, `"trigger.0"`, `"trigger.1.0"`, ‚Ä¶) computed from array indices at render time; every structural edit (add leaf, add group, remove, change combinator, change a leaf's var) triggers a full `renderQuestEditor()` re-render (paths are recomputed fresh each time ‚Äî simpler and more robust than trying to preserve DOM identity across array-length changes, at the cost of losing focus on structural edits; plain value typing does **not** re-render, so text/number entry keeps focus). Each group box has an All/Any `<select>` (swapping the object's `all`/`any` key while keeping the same child array) plus inline hint text stating the empty-group semantics; each leaf is a var-path `<select>` (dropdown-only, **never free-typed** ‚Äî options are grouped by namespace: Needs/Skills from `stats.json`, Economy (`funds`), Time (`time.hour`/`time.day`), Variables from `simstate.json`, Quests (`quests.<id>.state`, self-reference allowed per the brief) + an operator `<select>` (`gte`/`lte`/`eq`/`neq`) + an adaptive value input resolved by `describeVarPath()`: number input for needs/skills/funds/time, a `locked`/`active`/`done` `<select>` for quest-state paths, and for `vars.<name>` paths a control typed by that variable's own `type` (checkbox-as-select for boolean, number input for number, text otherwise). A stale/hand-edited var path that doesn't match any live option still renders (in an "Unknown (stale id)" `<optgroup>`) rather than silently snapping to a different value. Changing a leaf's var or operator resets its value to a type-appropriate default (`defaultValueFor`), never leaving a type-mismatched leftover (e.g. a string value under `gte`).
- **Rewards builder**: one row per `Reward`, a type `<select>` (`funds`/`setVar`/`unlockAsset`) that swaps in a fresh default reward object on type change, then type-specific fields ‚Äî `funds`: amount number input; `setVar`: variable dropdown (from `simstate.json`) + adaptive value input (same typing rule as condition leaves); `unlockAsset`: asset dropdown (from `assets.json`). Add/remove rows via `+ Add reward` / a per-row `√ó remove`.
- **Add quest**: name prompt ‚Üí `slugify` + `uniqueId` (identical helper pattern to every prior editor) ‚Üí `{ trigger: {all:[]}, completion: {all:[]}, rewards: [], onceOnly: true }`. The empty-`all` vacuous-TRUE-on-both-trigger-and-completion warning isn't a separate dialog ‚Äî it shows up immediately in the inline group hint text *and* the validation panel, which is arguably more useful (visible for the life of the quest, not just at creation).
- **Referential integrity ‚Äî both directions, same "warn with names, strip on confirm" policy as ¬ß2c/¬ß2d**:
  - **Delete a variable**: scans every quest's `trigger`/`completion` trees for a leaf with `var === "vars.<id>"` (via a generic `anyLeaf`/`stripCondition` recursive walker reused for both variables and quests) and every `rewards` array for a `setVar` entry with that `var`. Confirm dialog lists referencing quests by name and which part references it; on confirm, condition leaves are stripped (the containing `all`/`any` array is kept, even if now empty ‚Äî an emptied `all` becomes vacuously TRUE, an emptied `any` becomes FALSE, both stated in the dialog text) and matching `setVar` rewards are removed outright (there's no meaningful "null" reward to leave behind). Both `simstate.json` and `quests.json` are marked dirty and saved together.
  - **Delete a quest**: scans every *other* quest's conditions for a leaf `var === "quests.<id>.state"` (self-references are irrelevant since the quest and its own leaf disappear together). Same strip-on-confirm behavior, `quests.json` only (a quest reference never touches `simstate.json`).
  - `stripCondition(node, predicate)` mutates combinator arrays in place, recursing through arbitrarily nested `all`/`any`; it returns `null` only if the *root* condition is itself a stripped leaf (bare-leaf root is legal per the `Condition` union though the editor never produces one), and callers fall back to `{ all: [] }` in that case.
- **Validation panel**: recomputed on every render (`collectValidationIssues` ‚Üí `renderValidation`), covers (a) empty `all`/`any` groups with their exact runtime semantics spelled out, (b) unknown ids for every namespace (`needs.`/`skills.`/`vars.`/`quests.<id>.state`) plus unrecognized namespaces entirely, (c) a leaf with none of `gte`/`lte`/`eq`/`neq` set, (d) reward-side unknown ids (`setVar`'s var, `unlockAsset`'s asset) and a non-numeric `funds` amount. **Decision: validation never blocks saving** ‚Äî the header text says so explicitly ("warnings only, saving is never blocked"), because a designer mid-edit (e.g. a quest with an intentionally empty group while wiring up its conditions) shouldn't be locked out of saving their in-progress work; the dropdowns already prevent *new* unknown ids from being created; unknown ids only show up from hand-edited JSON or an id deleted elsewhere, and the panel's job is to surface that, not gate it.
- **Tests**: new `test/quest-editor.test.mjs` (modeled on `test/interaction-editor.test.mjs`) ‚Äî render (variable rows, quest sidebar incl. repeatable badge), variable CRUD (type/default edits incl. the null-default toggle, add+uniquify, both delete branches with exact stripped PUT payloads spanning both files), quest add+uniquify, a full condition-builder sequence building a nested `{all:[{var:funds,eq:500},{any:[{var:skills.english,gte:5}]}]}` tree and asserting the *exact* resulting JSON, leaf removal, all 3 reward types (add/edit/remove) with exact PUT payloads, quest delete both branches (referenced ‚Äî strips the other quest's condition leaf; unreferenced ‚Äî plain confirm), validation-panel assertions (unknown need/variable ids, missing operator, emptied group semantics), and sidebar search filtering. All pass; the full existing suite (`test/*.mjs`+`tools/*.mjs` via `node`, `test/*.ts`+`test/meshfit.test.mjs` via `npx tsx`) re-runs clean ‚Äî no regressions. `game/*.ts` untouched, so the strict type gate wasn't re-run (nothing to gate).

## 7. Locked designs ‚Äî designer requests of 2026-07-14

Four requests from the designer, specced here before implementation (build order in ¬ß4 items 6‚Äì9). Common thread: everything is data + tool UI per the ¬ß1 rule.

### 7.1 Doors as their own asset type

- **Doors become entries in `assets.json`** with `category: "door"` ‚Äî they reuse the whole asset machinery (mesh, footprint, editor) instead of a parallel system. A new general field `buyable: boolean` (default `true`, absent = true) is added to `AssetDef`; door entries ship `buyable: false` ‚Äî the designer doesn't want players buying/selling doors *yet*, but flipping one flag changes that later. The future Buy/Sell catalog must filter on `buyable`.
- **Mesh-fit fields, general to ALL assets** (any imported GLB may come mis-scaled/mis-oriented): optional `meshFit: { scale?: number | [x,y,z], yawOffsetDeg?: number, yOffset?: number }` on `AssetDef`, applied by the loader after normalization. This answers the door "resize as required" request but deliberately lands on every asset.
- **Door-specific block** on door-category assets: `door: { hingeOffset: [x, z], openAngleDeg, openSeconds, closeSeconds, triggerDistance }`. `hingeOffset` is the rotation-axis position in model-local meters relative to the model origin ‚Äî the designer's requested movable Z-rotation axis, so any GLB can be made to swing correctly regardless of where its author put the origin. Per-asset values override tuning defaults `tuning.doors: { openSeconds: 0.5, closeSeconds, openAngleDeg: 90, triggerDistance }` (all live-tunable; 0.5s is the designer's requested default).
- **Map link**: a map's `doors[]` entries gain an optional `assetId` (a door-category asset). With an `assetId` the game renders that asset's mesh on the doorway and animates it; without one, current behavior (bare opening) is unchanged ‚Äî old maps stay valid. The Map Editor's Doors mode gets a door-asset dropdown (door-category assets only) in its inspector.
- **Open/close logic** (mirrors the Unreal `BP_Door` semantics, which are PIE-proven): each frame-ish (reuse an existing tick, no new timer): if the sim is within `triggerDistance` **and** its current nav path crosses the doorway segment ‚Üí rotate open around the hinge over `openSeconds`; once the sim is farther than `triggerDistance`, rotate closed over `closeSeconds`. Never close while the sim is within the trigger distance (never trap/clip the sim mid-transit). Standing near a door without a path through it does NOT open it.

### 7.2 Asset facing direction (the "normal" vector)

- `AssetDef` gains `facingDeg: number` (default 0): the direction the object faces in **model-local space**, expressed as a yaw in degrees (0 = the model's local ‚àíZ after `meshFit.yawOffsetDeg`, i.e. what the game already treats as "toward the camera at rot 0" ‚Äî the implementing agent must verify and document the exact convention against `world.ts`'s placement math). An instance's **world facing** = instance `rot` + `facingDeg`. A single yaw is exactly the "normal vector telling me where the object is facing" the designer asked for, in the friendliest editable form (degrees, not a vector triple) ‚Äî tools may *display* it as an arrow.
- **Consumers (game)**: a shared `useSpotFor(instance)` helper ‚Äî the interaction stand/approach point becomes "footprint edge along world facing + small clearance" instead of any current pivot/center heuristic (fridge opens from its front, WC approached from the front‚Ä¶); seat-aware logic ("sit in front of the TV") screens candidate seats by the half-space test `dot(seatPos ‚àí tvPos, tvFacing) > 0` and the seat's own facing toward the target ‚Äî this replicates the Unreal Phase-A seat-in-front-of-screen logic that needed a hardcoded RightVector exception for the TV mesh; with `facingDeg` per asset that exception class disappears.
- **Tools**: Asset Editor gets the `facingDeg` field (and `meshFit`, and `buyable` from ¬ß7.1); Map Editor draws a small facing arrow on every placed object (read-only indicator, instance rot + asset facingDeg) so the designer can see at a glance which way furniture faces.

### 7.3 Accidents

- **Accident types are assets too**: `category: "accident"`, `buyable: false` (fire, water puddle, debris‚Ä¶). They get mesh/footprint/interactions through the existing Asset Editor for free. Their `interactions` are the cleanup actions (mop, extinguish ‚Äî authored in the Interaction Editor like any action); their `environmentScore` should typically be negative while present. New field on accident-category assets: `clearedBy: string[]` ‚Äî action ids whose completion on the accident instance despawns it.
- **Per-asset risk config** on any normal asset: `accidents: [{ accidentId, trigger: "onUse", baseChancePercent, placement: "on" | "adjacent", adjacentRange: [1, 2], modifiers: [{ var, pctAt0, pctAtMax }] }]`.
  - **Roll timing**: once, when a sim *finishes* an action on the asset (`trigger: "onUse"` is the only trigger for now; the enum exists so future triggers ‚Äî time-based, idle ‚Äî slot in).
  - **Risk formula**: `chance = clamp(baseChancePercent + Œ£ modifier contributions, 0, 100)`; each modifier reads a stat via the **quest condition namespace** (`needs.<id>`, `skills.<id>` ‚Äî reuse `game/quests.ts`'s path resolution, don't reinvent) and contributes a linear interpolation from `pctAt0` (stat at 0) to `pctAtMax` (stat at its max). Example: kitchen fire `baseChancePercent: 2`, modifier `{ var: "skills.cooking", pctAt0: +15, pctAtMax: ‚àí2 }` ‚Äî a novice cook risks ~17%, a master ~0%.
  - **Placement**: `"on"` spawns the accident on the base asset's own cells (kitchen fire ON the stove); `"adjacent"` picks a random free, in-bounds, non-wall cell 1‚Äì2 grid squares away (`adjacentRange`) ‚Äî the designer's "leak" case. If no free cell exists, fall back to "on".
- **Runtime**: accident *instances* are runtime state (spawned into the world + nav/tap systems like placed objects), NOT written into the map file; shape must be serializable for the future save system (same convention as `QuestRunner.serialize()`). 
- **Interaction hierarchy (the designer's exact rule)**: if an accident instance's footprint **overlaps** a base asset's footprint, the accident's interactions REPLACE the base asset's in the tap menu (impossible to cook while the kitchen is on fire) and autonomy must likewise skip the blocked asset's actions. If they do **not** overlap, there is no hierarchy ‚Äî the accident is just its own independent tappable object alongside the asset.
- **Tools**: the Asset Editor gains (a) nothing special for accident-category assets beyond the `clearedBy` action multi-select, and (b) an "Accidents" risk section on normal assets ‚Äî accident dropdown (accident-category assets only), trigger, base %, placement + range, modifier rows with var dropdowns fed from `stats.json` (no free-typed ids, as always).

### 7.4 Tool navigation

- A shared nav bar on every tool page + the game: new `tools/nav.js` (plain non-module script so jsdom tests are unaffected) that injects a compact tab strip ‚Äî Game, Assets, Interactions, Tuning, Map, Animations, Quests ‚Äî highlighting the current page. Each `tools/*.html` (and `index.html`) includes it with one `<script src>` line. New tools must add themselves to the one list in `nav.js`.
